# S8.2: Tipos Supabase + eliminar @ts-nocheck — Design

## Type Generation Approach

### Generated types (source of truth)

```bash
npx supabase gen types typescript --linked > src/integrations/supabase/types.ts
```

This produces a `Database` type with full schema: tables, views, functions, enums. The file is auto-generated and must not be edited manually.

### Typed client (frontend)

Update `src/integrations/supabase/client.ts`:

```typescript
import { createClient } from '@supabase/supabase-js';
import { Database } from './types';

export const supabase = createClient<Database>(SUPABASE_URL, SUPABASE_PUBLISHABLE_KEY);
```

This gives type inference on all `.from('table')` calls, `.select()`, `.insert()`, etc.

### Shared types for Edge Functions (Deno)

Create `supabase/functions/_shared/types.ts`:

```typescript
// Re-export Database type for Edge Functions
// Generated from: npx supabase gen types typescript --linked
export type { Database } from './database.types.ts';
```

The actual generated types file will be placed at `supabase/functions/_shared/database.types.ts` as a copy of the generated types, since Deno Edge Functions cannot import from `src/`. This is a deliberate duplication — both files come from the same `gen types` command.

## @ts-nocheck Removal Strategy

### Approach

1. Remove `// @ts-nocheck` from each function one at a time
2. Run `deno check` after each removal to find type errors
3. Fix type errors with minimal changes:
   - Add type annotations to variables and parameters
   - Replace `any` with proper types from `Database`
   - Use `as Type` casts for complex cases that would require major refactoring
4. DO NOT refactor business logic — only fix type errors
5. Commit after each function is clean

### Function order (simplest first)

| # | Function | Complexity | Notes |
|---|----------|-----------|-------|
| 1 | `send-message-v3` | Low | Few DB queries, straightforward message sending |
| 2 | `analyze-leads` | Medium | Uses configMap, multiple DB reads |
| 3 | `get-samurai-context` | Medium | Lots of string concatenation from DB data |
| 4 | `process-samurai-response` | High | Many DB operations, action parsing |
| 5 | `evolution-webhook` | Highest | Many branches, webhook payload parsing |

### Type error resolution rules

- **Prefer**: Proper type from `Database['public']['Tables']['table_name']['Row']`
- **Acceptable**: `as Type` cast when fixing would require refactoring logic
- **Avoid**: Replacing `// @ts-nocheck` with scattered `// @ts-ignore` (defeats the purpose)
- **Never**: `any` as a type annotation (use `unknown` if truly unknown)

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Generated types are auto-generated, never edit | Schema is source of truth; manual edits drift |
| Duplicate types for Deno (can't import from `src/`) | Deno runtime has separate module resolution |
| Fix order: simplest function first | Build confidence and patterns before tackling complex functions |
| `as Type` casts over `any` | Preserves type intent, documents assumptions, catches future drift |
| No business logic changes | Scope control — type safety only, no functional changes |

## Risks

| Risk | Mitigation |
|------|------------|
| Too many type errors in a single function | Time-box to 30min per function; if blocked, use targeted `as` casts |
| Generated types missing tables/views | Verify with `supabase db dump` that schema matches expectations |
| Deno import resolution issues with shared types | Test with `deno check` before committing each function |
