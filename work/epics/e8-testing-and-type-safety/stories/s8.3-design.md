# S8.3: Deno test + tests backend — Design

## Problem

All 5 target Edge Functions follow the same pattern:

```ts
import { serve } from "https://deno.land/std@0.190.0/http/server.ts"
serve(async (req: Request): Promise<Response> => {
  // CORS check
  // parse body
  // create supabase client
  // business logic
  // return Response
})
```

The `serve()` call happens at module level — importing the file starts an HTTP server. We cannot directly import and test the handler function.

## Solution: Mock `serve` to capture handler

The most pragmatic approach WITHOUT refactoring production code:

1. **Mock `serve`** before importing the function module. Replace `serve` with a stub that captures the handler callback instead of starting a server.
2. **Mock `createClient`** to return a fake Supabase client with chainable query methods.
3. **Mock `fetch`** globally to intercept OpenAI API calls and return configurable responses.
4. **Mock `Deno.env`** to provide required environment variables.
5. Call the captured handler with a crafted `Request`, assert on the `Response`.

This tests the full request-to-response flow including CORS, body parsing, error handling, and business logic — without touching production code.

## Test Helpers

### `_test_helpers/mock_supabase.ts`

Chainable mock that simulates Supabase client behavior:

```ts
// Usage:
const mock = createMockSupabase({
  leads: { select: [{ id: 'x', nombre: 'Test' }] },
  messages: { insert: { data: null, error: null } },
});
// mock.from('leads').select('*').eq('id', 'x').single()
// → returns { data: { id: 'x', nombre: 'Test' }, error: null }
```

Key features:
- Chainable: `.from().select().eq().single()` etc.
- Configurable per-table responses
- Tracks calls for assertions (which tables queried, what was inserted)
- Supports: `select`, `insert`, `update`, `upsert`, `delete`, `eq`, `neq`, `in`, `single`, `maybeSingle`, `order`, `limit`, `rpc`

### `_test_helpers/mock_fetch.ts`

Intercepts `globalThis.fetch` calls:

```ts
// Usage:
const restore = mockFetch({
  'https://api.openai.com/v1/chat/completions': {
    status: 200,
    body: { choices: [{ message: { content: 'AI response' } }] }
  }
});
// ... run test ...
restore(); // restore original fetch
```

Key features:
- URL-pattern matching for fetch interception
- Configurable status codes and response bodies
- Tracks calls for assertions
- Restore function for cleanup

### `_test_helpers/mock_serve.ts`

Captures the handler passed to `serve()`:

```ts
// Usage:
const { handler, cleanup } = await mockServe(
  '../send-message-v3/index.ts'
);
const response = await handler(new Request('http://localhost', {
  method: 'POST',
  body: JSON.stringify({ phone: '+521234567890', message: 'hello' })
}));
assertEquals(response.status, 200);
cleanup();
```

Implementation approach:
- Override the `serve` import in the module map so that when the function module loads, it calls our mock instead of the real `serve`
- The mock captures the handler and exposes it for direct invocation
- Alternative: use Deno's `--import-map` to redirect `serve` import

### `_test_helpers/setup.ts`

Common test setup:
- Set required `Deno.env` variables (`SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `OPENAI_API_KEY`)
- Provide default mock configurations
- Export test utilities (request builders, response parsers)

## Mock Strategy for `serve` — Implementation Detail

Since `serve` is imported from `https://deno.land/std@0.190.0/http/server.ts`, we have two options:

**Option A: Import map override** — Use `deno.json` with import map to redirect the `serve` import to our mock during tests. Requires a `deno.json` test config.

**Option B: Dynamic import with stub** — Set `globalThis.serve` before dynamic-importing the module. This does not work because `serve` is an ESM import, not a global.

**Option C (chosen): Pre-import stub via import map** — Create a `deno.json` in the test directory that maps `https://deno.land/std@0.190.0/http/server.ts` to a local mock. The mock's `serve` captures the handler. Tests use `deno test --config=deno.test.json`.

```jsonc
// supabase/functions/deno.test.json
{
  "imports": {
    "https://deno.land/std@0.190.0/http/server.ts": "./_test_helpers/mock_serve.ts",
    "https://esm.sh/@supabase/supabase-js@2.45.0": "./_test_helpers/mock_supabase.ts"
  }
}
```

This is the cleanest approach: zero production code changes, full control over mocks.

## Test Coverage Strategy

For each function, test at minimum:

| Category | Example |
|----------|---------|
| **Happy path** | Valid request → expected response |
| **CORS** | OPTIONS request → CORS headers |
| **Missing params** | No body / missing required fields → error response |
| **DB errors** | Supabase returns error → graceful handling |
| **Auth** | Missing env vars → appropriate failure |

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Import map to mock `serve` + `createClient` | Zero production code changes; mocks at import level |
| Test files colocated with source | Follows ADR-2 from E8 design |
| Separate `deno.test.json` config | Test imports don't affect production |
| Chainable mock over simple stubs | Matches real Supabase client API, less brittle |
| No refactoring of production code | Scope control — testing only |

## Risks

| Risk | Mitigation |
|------|------------|
| Import map may not intercept all imports | Verify with simplest function first (send-message-v3) |
| Dynamic imports in functions bypass import map | Functions use static imports — verified in Gemba |
| Mock Supabase client missing methods | Build incrementally, add methods as tests reveal needs |
| Deno version incompatibility | Deno 2.7.11 confirmed, supports import maps natively |
