# S8.1: Vitest Setup + Tests Frontend — Design

## Approach

### Framework Configuration

**Decision**: Separate `vitest.config.ts` file that imports from `vite.config.ts`.

ADR-1 recommends extending `vite.config.ts` with a `test` block. However, a separate
`vitest.config.ts` using `mergeConfig` is preferable because:
1. It keeps production config clean — no test-only keys in the build config
2. Vitest auto-discovers `vitest.config.ts` — no `defineConfig` collision
3. Path aliases are inherited from the Vite config via `mergeConfig`

```ts
// vitest.config.ts
import { defineConfig, mergeConfig } from 'vitest/config';
import viteConfig from './vite.config';

export default mergeConfig(viteConfig, defineConfig({
  test: {
    globals: true,
    environment: 'jsdom',
    setupFiles: ['./src/test/setup.ts'],
    include: ['src/**/*.test.ts', 'src/**/*.test.tsx'],
    coverage: {
      provider: 'v8',
      reporter: ['text', 'lcov'],
    },
  },
}));
```

**Rationale for jsdom over happy-dom**: jsdom is more mature, better documented,
wider community support. happy-dom is faster but has edge cases with complex DOM APIs.
For a project starting from zero tests, stability matters more than speed.

### Packages to Install (devDependencies)

| Package | Purpose |
|---------|---------|
| `vitest` | Test runner, assertion library, mocking |
| `jsdom` | DOM environment for component/hook tests |
| `@testing-library/react` | `renderHook`, React component test utilities |
| `@testing-library/jest-dom` | DOM matchers (`toBeInTheDocument`, etc.) |

No need for `@vitest/coverage-v8` in this story — coverage is out of scope.

### Path Alias Resolution

The `@/*` alias resolves automatically because `vitest.config.ts` merges the Vite
config which already defines `resolve.alias.@ = ./src`. No duplicate configuration needed.

### Test File Naming Convention

`<module>.test.ts` colocated next to the source file:
- `src/lib/tag-parser.test.ts`
- `src/lib/chat-normalizer.test.ts`
- `src/lib/utils.test.ts`
- `src/utils/logger.test.ts`
- `src/utils/messagingService.test.ts`
- `src/hooks/useRealtimeMessages.test.ts`

Colocation makes imports trivial (relative `./module`) and keeps test files discoverable
next to the code they cover.

## Test Strategy Per File

### 1. `src/lib/tag-parser.ts` — Pure logic, no mocking

**Functions**: `extractTagText()`, `parseTagsSafe()`

**Tests for `extractTagText`**:
- Returns `''` for null/undefined/empty input
- Returns string input unchanged
- Extracts `.text` from object
- Falls back to `.name`, `.label`, `.id` in order
- Returns `'Tag'` when object has no known fields
- Stringifies non-string, non-object values

**Tests for `parseTagsSafe`**:
- Returns `[]` for null/undefined/empty input
- Parses valid JSON string array of strings into tag objects
- Parses valid JSON string array of objects preserving id/text/color
- Returns `[]` for invalid JSON
- Returns `[]` for non-array parsed result
- Filters out null entries (non-string, non-object items)
- Generates fallback id as `tag-{idx}` when object has no id/text
- Accepts pre-parsed array (not just strings)

### 2. `src/lib/chat-normalizer.ts` — Pure logic, no mocking

**Function**: `normalizeLeadForChat()`

**Tests**:
- Returns full default object for null/undefined input
- Returns full default object for non-object input (string, number)
- Preserves all fields from a complete lead object
- Falls back `nombre` to `'Sin nombre'` when empty or missing
- Coerces `ai_paused` to boolean
- Defaults `tags` to `[]` when not an array
- Defaults `buying_intent` to `'BAJO'`
- Defaults `platform` to `'WHATSAPP'`
- Preserves optional fields (`estado`, `pais`, `cp`, `assigned_to`, `channel_id`)

### 3. `src/lib/utils.ts` — Pure logic, no mocking

**Function**: `cn()`

**Tests**:
- Merges multiple class strings
- Deduplicates conflicting Tailwind classes (e.g., `p-2` + `p-4` = `p-4`)
- Handles conditional classes via clsx syntax
- Returns empty string for no args

### 4. `src/utils/logger.ts` — Side effects, requires Supabase mock

**Function**: `logActivity()`

**Mocking strategy**: Mock the `@/integrations/supabase/client` module using
`vi.mock()`. The mock provides a chainable Supabase client:

```ts
vi.mock('@/integrations/supabase/client', () => ({
  supabase: {
    auth: { getUser: vi.fn() },
    from: vi.fn(() => ({
      select: vi.fn().mockReturnThis(),
      eq: vi.fn().mockReturnThis(),
      single: vi.fn(),
      insert: vi.fn(),
    })),
  },
}));
```

**Tests**:
- Inserts log with correct fields when user is authenticated
- Uses `'system'` as username when no user
- Uses profile username over email when available
- Does not throw on insert error (logs to console.error)
- Does not throw on critical error (logs to console.error)

### 5. `src/utils/messagingService.ts` — Side effects, requires Supabase mock

**Function**: `sendMessage()`

**Mocking strategy**: Same `vi.mock('@/integrations/supabase/client')` pattern.
The mock needs to handle chained `.from().select().eq().single()` and
`supabase.functions.invoke()`.

**Tests**:
- Uses `explicitChannelId` when provided (skips all lookups)
- Looks up channel from lead when no explicit channel
- Falls back to `app_config` default channel
- Falls back to first active `whatsapp_channels` entry
- Throws when no channel is found anywhere
- Calls `send-message-v3` edge function with correct payload
- Throws when edge function returns `success: false`
- `sendEvolutionMessage` is an alias for `sendMessage`

### 6. `src/hooks/useRealtimeMessages.ts` — React hook, needs renderHook + mocks

**Hook**: `useRealtimeMessages()`

**Mocking strategy**: Mock `@/integrations/supabase/client` with:
- `supabase.from().select().eq().order().then()` — for fetch and polling
- `supabase.channel().on().on().subscribe()` — for realtime
- `supabase.removeChannel()` — for cleanup

Use `@testing-library/react`'s `renderHook` + `act` + `waitFor`.

**Tests**:
- Returns empty messages and `loading: false` when `leadId` is null
- Returns empty messages and `loading: false` when `open` is false
- Fetches messages on mount when `leadId` is provided
- Sets `loading: false` after fetch completes
- Calls `refetch` to re-fetch messages
- Cleans up channel and interval on unmount

## Mocking Strategy (Shared)

All Supabase-dependent files import from the same path:
`@/integrations/supabase/client`

A single shared mock factory in `src/test/mocks/supabase.ts` provides a reusable
chainable mock. Each test file can import and customize it:

```ts
// src/test/mocks/supabase.ts
export function createMockSupabase() {
  const mockChain = {
    select: vi.fn().mockReturnThis(),
    eq: vi.fn().mockReturnThis(),
    single: vi.fn().mockResolvedValue({ data: null, error: null }),
    maybeSingle: vi.fn().mockResolvedValue({ data: null, error: null }),
    insert: vi.fn().mockResolvedValue({ error: null }),
    order: vi.fn().mockReturnThis(),
    limit: vi.fn().mockReturnThis(),
  };

  return {
    auth: { getUser: vi.fn().mockResolvedValue({ data: { user: null } }) },
    from: vi.fn(() => mockChain),
    functions: { invoke: vi.fn().mockResolvedValue({ data: { success: true }, error: null }) },
    channel: vi.fn(() => ({
      on: vi.fn().mockReturnThis(),
      subscribe: vi.fn(),
    })),
    removeChannel: vi.fn(),
  };
}
```

## Config Decisions Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Config file | `vitest.config.ts` (separate) | Clean separation, auto-discovery |
| DOM environment | `jsdom` | Stability over speed for initial setup |
| Test location | Colocated (`*.test.ts`) | Minimal import paths, easy discovery |
| Globals | `true` (`describe`, `it`, `expect` global) | Less boilerplate, Jest-familiar |
| Setup file | `src/test/setup.ts` | Import jest-dom matchers once |
| Mock pattern | `vi.mock()` + shared factory | DRY, consistent across test files |

| File | Change |
|------|--------|
| `package.json` | Add `"test": "vitest run"`, `"test:watch": "vitest"` to scripts |
| `tsconfig.app.json` | Add `vitest/globals` to `compilerOptions.types` (for global `describe`/`it`/`expect`) |

## No Production Source Changes

Per acceptance criteria, no source files (`src/lib/*.ts`, `src/utils/*.ts`,
`src/hooks/*.ts`) will be modified. Tests exercise the existing code as-is.
