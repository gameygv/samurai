# S9.3: Secrets to Env Vars — Design

## Migration Strategy

Two-phase approach for zero downtime:

1. **Phase A (this story)**: Update Edge Functions to read from env var with fallback to app_config:
   ```ts
   const apiKey = Deno.env.get('OPENAI_API_KEY') || configMap['openai_api_key'];
   ```
   This ensures that if the env var is not yet set, the function still works using the old app_config value.

2. **Phase B (post-deploy)**: After deploying functions AND running `supabase secrets set`, the env vars take priority. The app_config values become unused but remain as safety net.

3. **Phase C (future)**: After verification in production, remove secret rows from app_config. Out of scope for this story.

### META_ACCESS_TOKEN + META_PIXEL_ID (2 functions)

| Function | Current pattern | Change |
|----------|----------------|--------|
| analyze-leads | `configMap.meta_access_token`, `configMap.meta_pixel_id` | `Deno.env.get('META_ACCESS_TOKEN') \|\| configMap.meta_access_token` |
| meta-capi-sender | `config.access_token`, `config.pixel_id` from request body | Read from env var with fallback to request body (backward compat) |

### meta-capi-sender special case

Currently receives `pixel_id` and `access_token` from the request body (sent by Payments.tsx and analyze-leads). After this change:
- The function reads `META_ACCESS_TOKEN` and `META_PIXEL_ID` from env vars first
- Falls back to request body values for backward compatibility
- Frontend stops sending these values in the request body

```ts
// Before
const { pixel_id, access_token, test_event_code } = config;

// After
const pixel_id = Deno.env.get('META_PIXEL_ID') || config?.pixel_id;
const access_token = Deno.env.get('META_ACCESS_TOKEN') || config?.access_token;
const test_event_code = config?.test_event_code;
```

## Frontend Changes

### Payments.tsx
- Stop reading `meta_access_token` from app_config
- Stop sending `access_token` in the request body to meta-capi-sender
- Still send `pixel_id` temporarily (function reads from env with fallback)
- Remove: `config.meta_access_token` from the config object sent to the function

### MetaVerify.tsx
- Stop sending `accessToken` in request body
- The verification function reads token from env var
- UI still shows masked token status (reads from app_config for display only — or remove display)

## Env Vars to Document

```bash
supabase secrets set OPENAI_API_KEY="sk-..."
supabase secrets set META_ACCESS_TOKEN="EAA..."
supabase secrets set META_PIXEL_ID="..."
```

## Risk Mitigation

- **Fallback pattern**: Every function uses `Deno.env.get('X') || oldPattern` — if env var is missing, old behavior continues
- **Deploy order**: Set secrets BEFORE deploying new function code
- **No data loss**: app_config rows are NOT deleted in this story
- **Frontend backward compat**: meta-capi-sender accepts both env var and request body

## Decision: test_event_code stays in app_config

`test_event_code` and `test_mode` are configuration, not secrets. They remain in app_config and can be toggled from the UI.
