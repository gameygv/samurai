# S9.4: CORS Restrictivo + Hardening Final — Design

## CORS Strategy

### New `_shared/cors.ts`

The shared module exports a function `getCorsHeaders(req: Request)` plus a backward-compatible `corsHeaders` constant.

```ts
const ALLOWED_ORIGINS: string[] = [
  Deno.env.get('CORS_ALLOWED_ORIGIN') || '',
  'http://localhost:8080',
].filter(Boolean);

export function getCorsHeaders(req: Request): Record<string, string> {
  const origin = req.headers.get('Origin');

  // Server-to-server (no Origin header) — allow all (webhooks need this)
  if (!origin) {
    return {
      'Access-Control-Allow-Origin': '*',
      'Access-Control-Allow-Headers': CORS_ALLOWED_HEADERS,
    };
  }

  // Origin matches allowlist — echo it back
  if (ALLOWED_ORIGINS.includes(origin)) {
    return {
      'Access-Control-Allow-Origin': origin,
      'Access-Control-Allow-Headers': CORS_ALLOWED_HEADERS,
      'Vary': 'Origin',
    };
  }

  // Origin doesn't match — return first allowed origin (browser blocks mismatch)
  return {
    'Access-Control-Allow-Origin': ALLOWED_ORIGINS[0] || '',
    'Access-Control-Allow-Headers': CORS_ALLOWED_HEADERS,
    'Vary': 'Origin',
  };
}

// Backward-compatible constant — uses wildcard for functions not yet migrated
export const corsHeaders = {
  'Access-Control-Allow-Origin': '*',
  'Access-Control-Allow-Headers': CORS_ALLOWED_HEADERS,
};
```

### Allowed Headers

Unified set from all existing functions:
```
authorization, x-client-info, apikey, content-type, x-device-id
```

`x-device-id` is currently only in `send-message-v3` but adding it globally is harmless and avoids per-function overrides.

### Migration Pattern Per Function

**Before:**
```ts
const corsHeaders = {
  'Access-Control-Allow-Origin': '*',
  'Access-Control-Allow-Headers': 'authorization, x-client-info, apikey, content-type',
}
```

**After (simple — most functions):**
```ts
import { corsHeaders } from '../_shared/cors.ts'
```

Functions that handle OPTIONS preflight can optionally use `getCorsHeaders(req)` for origin-aware responses, but the constant `corsHeaders` with wildcard is sufficient for backward compatibility during migration.

### Ideal Migration (phase 2, future)

In a future pass, all functions should migrate from the static `corsHeaders` constant to `getCorsHeaders(req)` for full origin validation. This story focuses on unifying the import and adding the validation infrastructure.

## Decisions

- **D1**: Keep `corsHeaders` constant with `*` for backward compat — functions already importing it (23 functions) continue working without changes
- **D2**: New `getCorsHeaders(req)` is opt-in — functions can migrate gradually
- **D3**: `x-device-id` added to shared headers — no reason to restrict it per-function
- **D4**: No Origin = wildcard — server-to-server calls (webhooks, crons) don't send Origin headers
