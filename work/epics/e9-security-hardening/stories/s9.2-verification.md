# S9.2 Verification Queries

Run these queries in the Supabase SQL Editor after applying the migration.
Use "Impersonate user" in the Supabase dashboard or SET LOCAL to simulate roles.

## 1. Verify helper function exists

```sql
SELECT proname, pronamespace::regnamespace, prosecdef
FROM pg_proc
WHERE proname = 'get_user_role' AND pronamespace::regnamespace::text = 'auth';
-- Expected: 1 row, prosecdef = true
```

## 2. Verify all policies were created

```sql
SELECT schemaname, tablename, policyname, permissive, roles, cmd, qual
FROM pg_policies
WHERE schemaname = 'public'
ORDER BY tablename, policyname;
-- Expected: ~60 policies, no USING(true) policies remaining
```

## 3. Verify no USING(true) policies remain

```sql
SELECT tablename, policyname, qual
FROM pg_policies
WHERE schemaname = 'public'
  AND qual = 'true';
-- Expected: 0 rows
```

## 4. Verify anon has no table access

```sql
SELECT grantee, table_name, privilege_type
FROM information_schema.role_table_grants
WHERE grantee = 'anon' AND table_schema = 'public';
-- Expected: 0 rows
```

## 5. Test as ADMIN — should see all leads

```sql
-- Replace with a real admin UUID from profiles
SET LOCAL role = 'authenticated';
SET LOCAL request.jwt.claims = '{"sub": "<ADMIN_UUID>"}';

SELECT count(*) FROM leads;
-- Expected: all leads
SELECT count(*) FROM app_config WHERE category = 'SECRET';
-- Expected: > 0 (admin sees secrets)
```

## 6. Test as AGENT — should see only assigned leads

```sql
-- Replace with a real agent UUID from profiles
SET LOCAL role = 'authenticated';
SET LOCAL request.jwt.claims = '{"sub": "<AGENT_UUID>"}';

SELECT count(*) FROM leads;
-- Expected: only leads where assigned_to = AGENT_UUID

SELECT count(*) FROM conversaciones;
-- Expected: only conversations for assigned leads

SELECT * FROM app_config WHERE category = 'SECRET';
-- Expected: 0 rows (agent cannot see secrets)

SELECT count(*) FROM profiles;
-- Expected: 1 (own profile only)
```

## 7. Test agent INSERT on activity_logs

```sql
SET LOCAL role = 'authenticated';
SET LOCAL request.jwt.claims = '{"sub": "<AGENT_UUID>"}';

INSERT INTO activity_logs (username, action, resource, description, status)
VALUES ('test_agent', 'TEST', 'RLS', 'Verification insert', 'OK');
-- Expected: success

SELECT * FROM activity_logs WHERE username = 'test_agent';
-- Expected: 0 rows (agent cannot SELECT logs)

-- Clean up (run as admin):
DELETE FROM activity_logs WHERE username = 'test_agent';
```

## 8. Test GERENTE — full read, restricted secrets

```sql
SET LOCAL role = 'authenticated';
SET LOCAL request.jwt.claims = '{"sub": "<GERENTE_UUID>"}';

SELECT count(*) FROM leads;
-- Expected: all leads

SELECT count(*) FROM app_config WHERE category = 'SECRET';
-- Expected: 0 rows (gerente cannot see secrets)

SELECT count(*) FROM app_config WHERE category != 'SECRET';
-- Expected: > 0 (gerente sees non-secret config)
```

## 9. Test ANON — should see nothing

```sql
SET LOCAL role = 'anon';

SELECT * FROM leads;
-- Expected: 0 rows or permission denied
```

## 10. Test credit chain for agent

```sql
SET LOCAL role = 'authenticated';
SET LOCAL request.jwt.claims = '{"sub": "<AGENT_UUID>"}';

SELECT count(*) FROM credit_sales;
-- Expected: only sales linked to contacts of assigned leads

SELECT count(*) FROM credit_installments;
-- Expected: only installments linked to own credit sales
```

## 11. Verify idx_leads_assigned_to exists

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'leads' AND indexname = 'idx_leads_assigned_to';
-- Expected: 1 row
```

## 12. Verify Edge Functions still work (service_role bypass)

Test by triggering an Edge Function (e.g., evolution-webhook) and confirming
it can INSERT into conversaciones and activity_logs. service_role bypasses
RLS entirely, so no policy changes should affect Edge Functions.
