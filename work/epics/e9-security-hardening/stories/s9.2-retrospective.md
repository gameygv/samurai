# S9.2 Retrospective: RLS Policies Reales por Rol

## Summary

Replaced all permissive `USING(true)` RLS policies across 21 tables with real role-based access control. Created `auth.get_user_role()` helper function, forward migration (56 new policies), and rollback migration. Architecture Review and Quality Review both passed with findings that were fixed before close.

## What Went Well

- **Comprehensive 21-table coverage**: Every table with RLS enabled got role-specific policies. No table left with `USING(true)`.
- **Fail-closed design**: `REVOKE ALL FROM anon` ensures zero access without a valid JWT. Agent policies default-deny unless `assigned_to = auth.uid()` matches.
- **Rollback ready**: Full rollback migration mirrors forward 1:1, restoring original permissive policies and anon grants in a single transaction.
- **Clean TDD-style progression**: T1 (helper) -> T2 (forward) -> T3 (rollback) -> T4 (verify build) -> T5 (verification queries). Each task committed independently.
- **SECURITY DEFINER pattern**: `auth.get_user_role()` cleanly avoids the circular RLS dependency on profiles without exposing the function to abuse.
- **Performance-aware**: Added `idx_leads_assigned_to` index for agent subquery performance. STABLE function caching avoids repeated profile lookups.

## What Could Improve

- **AR caught missing `profiles_agent_update` policy**: The design specified agent SELECT-only on profiles, but the frontend profile edit flow needs UPDATE on the agent's own row. AR question Q2 identified this gap; fixed in commit `cbf52f2`.
- **QR caught rollback gap**: The rollback migration was missing the `DROP POLICY IF EXISTS "profiles_agent_update"` statement added in the AR fix. Fixed in commit `4f895b7`.
- **Lesson**: Review gates (AR/QR) earned their keep. Both caught real issues that would have caused problems in production.

## Patterns Learned

1. **Always verify frontend writes against RLS policies** — design specified read-only for agents on profiles, but the frontend updates the user's own profile. Cross-referencing frontend operations against the policy matrix is essential.
2. **Rollback must mirror forward 1:1** — any policy added post-design (like `profiles_agent_update`) must also be added to the rollback DROP list. Rollback is not "write once and forget."
3. **SECURITY DEFINER helper functions simplify policy authoring** — without `auth.get_user_role()`, every policy would need a JOIN to profiles, creating circular dependencies and verbose SQL.
