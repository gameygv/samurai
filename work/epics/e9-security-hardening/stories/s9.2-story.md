# S9.2: RLS policies reales por rol

## Objective

Replace all `USING(true)` RLS policies with role-based policies that enforce data isolation by user role.

## What

Create a Supabase migration that drops every permissive `USING(true)` policy and replaces it with role-aware policies across all core tables: leads, conversaciones, contacts, profiles, media_assets, whatsapp_channels, app_config, credit_sales, credit_installments.

## Access Model

| Role | Access |
|------|--------|
| `admin` / `dev` | Full read/write on all tables |
| `gerente` | Full read on all tables (manager view) |
| `agent` | Read/write only on data assigned to them (`assigned_to = auth.uid()`) |
| `service_role` | Bypasses RLS (Edge Functions, crons) — no change needed |

## Key Rules

- Agent can only SELECT leads where `assigned_to = auth.uid()`
- Agent can only see conversaciones for their assigned leads
- Agent cannot read `app_config` rows with `category = 'SECRET'`
- A rollback migration must exist to restore previous policies if something breaks
- Revoke `GRANT ALL` from `anon` role (undo `FIX_PERMISSIONS.sql` damage)

## Risk

High. A misconfigured RLS policy breaks the entire app for all users. Testing with real role queries is mandatory before deploy. Rollback SQL must be ready.

## Dependencies

None (independent of S9.1), but S9.1 validates the auth pattern that informs testing.
