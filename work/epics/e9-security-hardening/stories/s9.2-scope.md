# S9.2 Scope: RLS policies reales por rol

## Objective

Replace permissive `USING(true)` RLS policies with real role-based access control. Agents see only their assigned data, admins/devs see everything, gerentes have full read access.

## In Scope

### Tables to secure
- `leads` — agent sees only `assigned_to = auth.uid()`
- `conversaciones` — agent sees only conversations linked to their leads
- `contacts` — agent sees only contacts linked to their leads
- `profiles` — all authenticated users can read; only admin can write
- `media_assets` — agent sees only assets linked to their leads/conversations
- `whatsapp_channels` — all authenticated users can read; only admin can write
- `app_config` — agent cannot read rows with `category = 'SECRET'`
- `credit_sales` — agent sees only sales for their leads
- `credit_installments` — agent sees only installments for their leads' sales

### Additional actions
- Revoke `GRANT ALL` from `anon` role (undo FIX_PERMISSIONS.sql)
- Create rollback migration to restore previous policies

## Out of Scope

- Storage policies (bucket-level RLS)
- Auth schema changes
- Edge Function code changes
- Custom JWT claims
- Rate limiting

## Acceptance Criteria

1. Agent can only SELECT leads where `assigned_to = auth.uid()`
2. Agent can only see conversaciones for their assigned leads
3. Agent cannot read `app_config` rows with `category = 'SECRET'`
4. Admin/dev can read/write all tables without restriction
5. Gerente can read all data across all tables (manager view)
6. `service_role` (Edge Functions, crons) bypasses RLS as before — no breakage
7. A rollback migration exists that restores previous `USING(true)` policies
8. `GRANT ALL` revoked from `anon` role
