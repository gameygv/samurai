# S9.2 Design: RLS Policies Reales por Rol

## Context

Every RLS policy in SAMURAI is `USING(true)` — no data isolation exists. `FIX_PERMISSIONS.sql` grants `ALL ON ALL TABLES` to `anon`, making the anon key equivalent to service_role for data access. This story replaces all permissive policies with real role-based access control.

## ADR Reference

This design implements **ADR-2** from the epic design (`work/epics/e9-security-hardening/design.md`): RLS policies use `auth.uid()` for user filtering and joins to `profiles.role` for role-based decisions.

---

## Helper Function: `auth.get_user_role()`

A PostgreSQL function to avoid repeating the profiles join in every policy.

```sql
CREATE OR REPLACE FUNCTION auth.get_user_role()
RETURNS TEXT
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = public
AS $$
  SELECT role FROM public.profiles WHERE id = auth.uid()
$$;
```

**Why SECURITY DEFINER**: The function runs as the definer (postgres), so it can read `profiles` even if the calling user's RLS would restrict it. Without this, a circular dependency occurs: the policy needs the role, but reading the role requires passing a policy.

**Why STABLE**: The role doesn't change within a transaction. PostgreSQL can cache the result per-statement, avoiding repeated lookups.

### Role Grouping

Policies use `auth.get_user_role()` to determine the caller's role:
- **admin_full**: `role IN ('admin', 'dev')` — unrestricted
- **manager_full**: `role = 'gerente'` — full read, write on most tables
- **agent_scoped**: `role IN ('agent', 'sales_agent', 'sales')` — scoped to owned data

## Policy Details (Key Tables)

### leads

```sql
-- Admin/dev: full access
CREATE POLICY "leads_admin_all" ON public.leads
FOR ALL TO authenticated
USING (auth.get_user_role() IN ('admin', 'dev'))
WITH CHECK (auth.get_user_role() IN ('admin', 'dev'));

-- Gerente: full access
CREATE POLICY "leads_gerente_all" ON public.leads
FOR ALL TO authenticated
USING (auth.get_user_role() = 'gerente')
WITH CHECK (auth.get_user_role() = 'gerente');

-- Agent: only assigned leads
CREATE POLICY "leads_agent_select" ON public.leads
FOR SELECT TO authenticated
USING (
  auth.get_user_role() IN ('agent', 'sales_agent', 'sales')
  AND assigned_to = auth.uid()
);

CREATE POLICY "leads_agent_update" ON public.leads
FOR UPDATE TO authenticated
USING (
  auth.get_user_role() IN ('agent', 'sales_agent', 'sales')
  AND assigned_to = auth.uid()
)
WITH CHECK (
  auth.get_user_role() IN ('agent', 'sales_agent', 'sales')
  AND assigned_to = auth.uid()
);

CREATE POLICY "leads_agent_insert" ON public.leads
FOR INSERT TO authenticated
WITH CHECK (
  auth.get_user_role() IN ('agent', 'sales_agent', 'sales')
  AND assigned_to = auth.uid()
);
```

### credit_sales (3-hop chain)

```sql
-- Agent access via: credit_sales.contact_id → contacts.lead_id → leads.assigned_to
CREATE POLICY "credit_sales_agent_select" ON public.credit_sales
FOR SELECT TO authenticated
USING (
  auth.get_user_role() IN ('agent', 'sales_agent', 'sales')
  AND contact_id IN (
    SELECT c.id FROM public.contacts c
    JOIN public.leads l ON c.lead_id = l.id
    WHERE l.assigned_to = auth.uid()
  )
);
```

### credit_installments (4-hop chain)

```sql
-- Agent access via: installments.sale_id → credit_sales.contact_id → contacts.lead_id → leads.assigned_to
CREATE POLICY "installments_agent_select" ON public.credit_installments
FOR SELECT TO authenticated
USING (
  auth.get_user_role() IN ('agent', 'sales_agent', 'sales')
  AND sale_id IN (
    SELECT cs.id FROM public.credit_sales cs
    JOIN public.contacts c ON cs.contact_id = c.id
    JOIN public.leads l ON c.lead_id = l.id
    WHERE l.assigned_to = auth.uid()
  )
);
```

### whatsapp_channels

```sql
-- Admin/dev: full access
CREATE POLICY "channels_admin_all" ON public.whatsapp_channels
FOR ALL TO authenticated
USING (auth.get_user_role() IN ('admin', 'dev'))
WITH CHECK (auth.get_user_role() IN ('admin', 'dev'));

-- Gerente: read all
CREATE POLICY "channels_gerente_select" ON public.whatsapp_channels
FOR SELECT TO authenticated
USING (auth.get_user_role() = 'gerente');

-- Agent: read basic info only (column-level restriction via view or selective grant)
-- Note: RLS cannot restrict columns. Two options:
--   (a) Create a view `whatsapp_channels_public` with only id, name, is_active
--   (b) REVOKE SELECT on sensitive columns from authenticated, then re-GRANT on safe columns
-- Decision: Use option (b) — column-level GRANT. See Migration Strategy below.
CREATE POLICY "channels_agent_select" ON public.whatsapp_channels
FOR SELECT TO authenticated
USING (auth.get_user_role() IN ('agent', 'sales_agent', 'sales'));
```

Column-level grant for agents on whatsapp_channels:
```sql
-- After revoking the blanket GRANT ALL:
GRANT SELECT (id, name, is_active, phone_number) ON public.whatsapp_channels TO authenticated;
-- Admin/dev/gerente policies allow full SELECT, but column grants limit what authenticated sees.
```

**Problem**: Column-level grants apply to the role, not per-user. Since admin/dev/gerente are all `authenticated`, we cannot use column grants to differentiate. **Revised decision**: Accept that agents can SELECT all columns via RLS policy (the api_key is the concern). Secret columns like `channel_token` and `api_key` should be migrated to Supabase Secrets in S9.3 (same pattern as openai_api_key). For S9.2, agents get full read on whatsapp_channels — the real protection is removing secrets from the table in S9.3.

## Migration Strategy

### File 1: Forward Migration (`20260407_rls_real_policies.sql`)

Execution order within the migration:

1. **Create helper function** `auth.get_user_role()`
2. **Drop ALL existing policies** on every table with RLS enabled
3. **Create new role-based policies** per table (as detailed above)
4. **Revoke anon grants**: `REVOKE ALL ON ALL TABLES IN SCHEMA public FROM anon`
5. **Grant minimal anon access**: None (anon should not access any table — all access goes through authenticated JWT)
6. **Grant authenticated base access**: `GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO authenticated` (RLS policies control the actual access)
7. **Notify PostgREST**: `NOTIFY pgrst, 'reload config'` and `NOTIFY pgrst, 'reload schema'`

The entire migration runs in a single transaction (`BEGIN; ... COMMIT;`).

### File 2: Rollback Migration (`20260407_rls_rollback.sql`)

Restores the previous state:

1. **Drop new helper function** `auth.get_user_role()`
2. **Drop ALL new policies** on every table
3. **Recreate original `USING(true)` policies** (exact copies from current SQL files)
4. **Re-grant anon access**: `GRANT ALL ON ALL TABLES IN SCHEMA public TO anon` (restores FIX_PERMISSIONS.sql behavior)
5. **Notify PostgREST**: reload config/schema

## Key Decisions

1. **`auth.get_user_role()` as SECURITY DEFINER** — avoids circular RLS dependency. The function reads profiles bypassing RLS, returns the role string. Every policy calls this once per statement (STABLE caching).

2. **Agents use `assigned_to` for lead access** — the `leads.assigned_to` column (UUID FK to profiles.id) determines ownership. Agent policies filter by `assigned_to = auth.uid()`.

3. **Contacts access via `lead_id`** — `contacts.lead_id` links to leads, so agent access is `lead_id IN (SELECT id FROM leads WHERE assigned_to = auth.uid())`.

4. **Credit chain is 3-4 hops** — `credit_sales.contact_id → contacts.lead_id → leads.assigned_to` and `credit_installments.sale_id → credit_sales → contacts → leads`. Subquery in USING clause handles this.

5. **`app_config` secrets filtered by category** — non-admin roles get `category != 'SECRET'` filter. After S9.3 migrates secrets to env vars, the SECRET rows will be deleted entirely.

6. **`whatsapp_channels` column restriction deferred to S9.3** — RLS cannot restrict columns. The real fix is removing api_key/channel_token from the table (S9.3 scope). For S9.2, agents can read the full row but can only SELECT (no write).

7. **Agent INSERT on conversaciones and activity_logs** — agents need to send messages (creates conversaciones rows) and log actions. INSERT policies use the same ownership check for conversaciones; activity_logs allows INSERT without restriction.

8. **Anon role fully revoked** — `REVOKE ALL FROM anon` on public schema. The anon key without a JWT gets zero access. Edge Functions use service_role_key (bypasses RLS). Frontend uses anon key with user JWT (becomes authenticated).

9. **Rollback ready** — `20260407_rls_rollback.sql` restores every original policy and re-grants anon access. Can be run in < 1 second to recover from a bad deploy.

10. **Single transaction** — the forward migration is wrapped in `BEGIN/COMMIT`. If any statement fails, nothing changes.

## Performance Considerations

- **`auth.get_user_role()` caching**: Marked `STABLE`, PostgreSQL evaluates it once per statement. For pages loading multiple tables, this means one profiles lookup per query, not per row.
- **Subquery in agent policies**: `lead_id IN (SELECT id FROM leads WHERE assigned_to = auth.uid())` runs as a semi-join. With an index on `leads.assigned_to`, this is fast even with thousands of leads.
- **Index requirement**: `CREATE INDEX IF NOT EXISTS idx_leads_assigned_to ON public.leads(assigned_to)` — must exist for agent policy performance. Add in the forward migration.
- **Credit chain subqueries**: 3-4 hop joins run once per query, not per row (PostgreSQL optimizes IN subqueries). With proper FK indexes, performance is acceptable.

## Testing Strategy

Before deploying to production, verify with direct SQL queries:

```sql
-- Test as admin (should see all)
SET LOCAL role = 'authenticated';
SET LOCAL request.jwt.claim.sub = '<admin_uuid>';
SELECT count(*) FROM leads;  -- expect: all leads

-- Test as agent (should see only assigned)
SET LOCAL request.jwt.claim.sub = '<agent_uuid>';
SELECT count(*) FROM leads;  -- expect: only assigned leads

-- Test as agent reading secrets (should see none)
SELECT * FROM app_config WHERE category = 'SECRET';  -- expect: 0 rows

-- Test as anon (should fail)
SET LOCAL role = 'anon';
SELECT * FROM leads;  -- expect: 0 rows or permission denied
```

## Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Policy typo breaks all access | Transaction wraps entire migration; rollback SQL ready |
| Agent can't see their leads after deploy | Test with real agent UUID before going live |
| Edge Functions break | They use service_role_key — bypass RLS. No impact. |
| Crons break | They use service_role_key — bypass RLS. No impact. |
| Performance degradation from subqueries | Add `idx_leads_assigned_to` index; STABLE function caching |
| Circular RLS on profiles | `auth.get_user_role()` is SECURITY DEFINER — bypasses profiles RLS |
