# S9.1 Retrospective — Auth middleware + proteger funciones admin

## Summary

Created `_shared/auth.ts` with `requireAuth()` and `requireRole()` helpers. Integrated auth gate in 6 admin Edge Functions. Architecture review found 2 issues (info leak in 403 responses, fail-open on missing profile); both fixed before merge.

## What Went Well

- **Clean integration pattern**: The 3-line auth gate (import + requireRole + early return) applied identically to all 6 functions with zero changes to existing business logic.
- **Design-first paid off**: The design doc specified exact code, exact integration points, and exact files. Implementation was mechanical — no ambiguity, no rework.
- **AR caught real issues**: Architecture review found an information leak (role value exposed in 403 error message) and a fail-open default (missing profile returned generic error instead of denying access). Both were legitimate security defects caught before merge.
- **Existing tests unaffected**: 56 frontend tests continued passing — auth changes were isolated to Edge Functions with no frontend coupling.

## What Could Improve

- **AR findings should have been caught in design**: The info leak (exposing `profile.role` in error messages) and the fail-open on missing profile were both foreseeable during design. Future designs should include an "error response audit" section.
- **No automated Edge Function tests**: The 6 admin functions lack automated integration tests. Manual curl verification works but doesn't prevent regression. Deferred to E8 testing scope expansion.

## Patterns Learned

### PAT: Sanitize Error Responses (from AR finding)

**Context**: Auth middleware returning detailed error info.
**Problem**: Original 403 response included `Forbidden: insufficient role (current: {role})` — leaking the user's actual role to the caller.
**Resolution**: Changed to generic `Forbidden: insufficient role` with no role value.
**Rule**: Never include internal state (roles, IDs, stack traces) in client-facing error responses.

### PAT: Fail-Closed on Missing Data (from AR finding)

**Context**: Profile lookup in `requireAuth()` when profile row doesn't exist.
**Problem**: Original code returned a 401 with "User profile not found" — this is fail-open behavior because it tells the attacker the JWT is valid but the profile is missing.
**Resolution**: Return generic "Invalid or expired token" (same as invalid JWT) so attackers can't distinguish between invalid tokens and missing profiles.
**Rule**: Auth failures should be indistinguishable from the attacker's perspective. Never reveal which step failed.

## Carry-Forward

- **Edge Function integration tests**: No automated tests for auth gate behavior (401/403/200). Should be addressed when Deno test infrastructure is set up (E8 expansion or future story).
- **CORS on error responses**: Auth error responses omit CORS headers. Deferred to S9.4 (CORS hardening).
- **Rate limiting**: Auth gate validates tokens but doesn't rate-limit failed attempts. Consider for future security hardening.
