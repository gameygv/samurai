# S9.1 Design — Auth middleware + proteger funciones admin

## Context

All 6 admin Edge Functions (system-wipe, admin-create-user, admin-reset-password, manage-auth-users, update-user-email, manage-prompt-versions) currently accept any HTTP request without verifying the caller's identity or role. This design specifies the `_shared/auth.ts` helper module and the integration pattern for each function.

## Gemba (current state of the 6 functions)

Every function follows the same structure:
1. Import `serve`, `createClient`, `corsHeaders`
2. Handle OPTIONS (CORS preflight)
3. Create `supabaseAdmin` with `SUPABASE_SERVICE_ROLE_KEY`
4. Parse body, do work, return JSON

No function reads the `Authorization` header. The only "security" is `system-wipe` checking for `confirmation === 'FACTORY_RESET'` in the body.

## Decision: `_shared/auth.ts` helper (implements ADR-1)

### Module exports

```ts
// _shared/auth.ts

import { createClient } from 'https://esm.sh/@supabase/supabase-js@2.45.0'

interface AuthResult {
  user: any
  profile: { id: string; role: string; full_name: string }
}

interface AuthResponse {
  data: AuthResult | null
  error: Response | null
}

/**
 * Extract JWT from Authorization header, validate it, return user.
 * Uses ANON key client for JWT validation (RLS-aware, not service_role).
 * Uses service_role client for profile lookup (bypasses RLS).
 */
export async function requireAuth(req: Request): Promise<AuthResponse> {
  const authHeader = req.headers.get('Authorization')
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return {
      data: null,
      error: new Response(
        JSON.stringify({ error: 'Missing or invalid Authorization header' }),
        { status: 401, headers: { 'Content-Type': 'application/json' } }
      )
    }
  }

  const token = authHeader.replace('Bearer ', '')

  // Validate JWT with ANON key — this respects RLS and validates the token
  // without granting service_role privileges during validation
  const supabaseAuth = createClient(
    Deno.env.get('SUPABASE_URL') ?? '',
    Deno.env.get('SUPABASE_ANON_KEY') ?? ''
  )

  const { data: { user }, error: userError } = await supabaseAuth.auth.getUser(token)
  if (userError || !user) {
    return {
      data: null,
      error: new Response(
        JSON.stringify({ error: 'Invalid or expired token' }),
        { status: 401, headers: { 'Content-Type': 'application/json' } }
      )
    }
  }

  // Look up profile.role with service_role client (bypasses RLS to always find the row)
  const supabaseAdmin = createClient(
    Deno.env.get('SUPABASE_URL') ?? '',
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY') ?? ''
  )

  const { data: profile, error: profileError } = await supabaseAdmin
    .from('profiles')
    .select('id, role, full_name')
    .eq('id', user.id)
    .single()

  if (profileError || !profile) {
    return {
      data: null,
      error: new Response(
        JSON.stringify({ error: 'User profile not found' }),
        { status: 401, headers: { 'Content-Type': 'application/json' } }
      )
    }
  }

  return { data: { user, profile }, error: null }
}

/**
 * Calls requireAuth, then checks if profile.role is in allowedRoles.
 * Returns 403 if role not allowed.
 */
export async function requireRole(
  req: Request,
  allowedRoles: string[]
): Promise<AuthResponse> {
  const result = await requireAuth(req)
  if (result.error) return result

  const { profile } = result.data!
  if (!allowedRoles.includes(profile.role)) {
    return {
      data: null,
      error: new Response(
        JSON.stringify({ error: 'Forbidden: insufficient role' }),
        { status: 403, headers: { 'Content-Type': 'application/json' } }
      )
    }
  }

  return result
}
```

### Key design decisions

1. **ANON key for JWT validation** — `auth.getUser(token)` sends the token to Supabase Auth server for validation. Using the ANON key ensures we don't accidentally grant service_role privileges during the validation step. The token itself proves the user's identity.

2. **Service_role for profile lookup** — After validating the user, we need their `profiles.role`. Since RLS policies may change (S9.2), using service_role guarantees we always find the profile row regardless of RLS state.

3. **Return `{ data, error }` pattern** — Matches Supabase SDK convention. The `error` field is a ready-to-return `Response` object, so the calling function can do `if (authError) return authError` with zero ceremony.

4. **No supabaseClient parameter** — The helper creates its own clients internally. This keeps the call site minimal (just `req` and `roles`) and avoids threading client references through function signatures.

5. **CORS headers not included in error responses** — Auth error responses omit CORS headers intentionally. The calling function handles CORS at the OPTIONS level. If CORS headers are needed on error responses, each function can add `corsHeaders` to the spread. This is deferred to S9.4 (CORS hardening).

## Integration pattern

Each admin function adds 3 lines after the CORS check:

```ts
import { requireRole } from '../_shared/auth.ts'

serve(async (req) => {
  if (req.method === 'OPTIONS') {
    return new Response(null, { headers: corsHeaders })
  }

  // === NEW: Auth gate (3 lines) ===
  const auth = await requireRole(req, ['admin', 'dev'])
  if (auth.error) return auth.error
  // auth.data.user and auth.data.profile available if needed
  // === END auth gate ===

  try {
    // ... existing function body unchanged ...
  }
})
```

Tests will validate the auth helper and integration using Deno test runner:

1. **RED**: `requireAuth` returns 401 when no Authorization header present
2. **RED**: `requireAuth` returns 401 when token is invalid/expired
3. **RED**: `requireRole` returns 403 when user role not in allowedRoles
4. **GREEN**: `requireRole` returns user+profile when role matches
5. **Integration**: `system-wipe` returns 401 without JWT (curl test)
6. **Integration**: `system-wipe` returns 403 with agent JWT (curl test)
7. **Smoke**: All 9 exempt functions still work without auth headers

## Risks

| Risk | Mitigation |
|------|-----------|
| `SUPABASE_ANON_KEY` not available in Edge Functions | It's a default env var in Supabase Edge Functions — verify in deploy |
| Profile lookup adds latency (~50ms) per request | Acceptable for admin functions (low traffic). Cache if needed later |
| Existing frontend callers don't send JWT | `supabase.functions.invoke()` automatically includes the user's JWT — no frontend changes needed |
