# S2.3: Design — Procesar status webhooks de Meta

## Change

### evolution-webhook/index.ts

Replace the early return on `change.statuses` (line 34) with status processing:

```
Current: if (!change || change.statuses) return new Response('ok', { status: 200 });
New:     if (!change) return new Response('ok', { status: 200 });
         if (change.statuses) → process status updates
```

Status processing logic:
1. Extract `status.id` (wamid) and `status.status` from payload
2. UPDATE conversaciones SET delivery_status = status WHERE message_id = wamid
3. If no row updated (wamid not found), ignore silently (D1)
4. If status is 'failed', also log to activity_logs with error details (D2)
5. Return 200 immediately (no heavy processing)

## Design Decisions

| ID | Decision | Choice | Rationale |
|----|----------|--------|-----------|
| S2.3-D1 | Status without wamid match | Ignore silently | Legacy messages self-resolve, avoid log noise |
| S2.3-D2 | Failed status handling | Save + log to activity_logs | Rare but important, operator needs visibility |

## Not Changed

- send-message-v3 — no changes
- process-samurai-response — no changes
- Frontend — delivery_status display is future work
- Database — columns exist from S2.2
