# S2.2: Design — Deduplicación por wamid en conversaciones

## Migration

Add two columns + partial unique index via Supabase CLI migration:

```sql
ALTER TABLE conversaciones ADD COLUMN message_id TEXT;
ALTER TABLE conversaciones ADD COLUMN delivery_status TEXT;
CREATE UNIQUE INDEX idx_conversaciones_message_id
  ON conversaciones (message_id) WHERE message_id IS NOT NULL;
```

Apply with `supabase db push`.

## Changes

### 1. send-message-v3/index.ts

- Parse Meta response as JSON (not raw text)
- Extract `messages[0].id` as wamid for provider `meta`
- Return `{ success, data, wamid }` (wamid is null for non-meta providers)
- Update API version: v20.0 → v21.0 (S2.4 bundled)

### 2. evolution-webhook/index.ts

- Pass `messageId` (already extracted at line 42) to conversaciones insert for CLIENTE messages
- Use ON CONFLICT (message_id) WHERE message_id IS NOT NULL DO NOTHING for dedup

### 3. process-samurai-response/index.ts

- Reorder: send to WhatsApp FIRST → get wamid → THEN insert in conversaciones with wamid
- If send fails, log error but do NOT insert (message never reached customer)
- If send succeeds but wamid is null (non-meta provider), insert without wamid

## Design Decisions

| ID | Decision | Choice | Rationale |
|----|----------|--------|-----------|
| S2.2-D1 | Migration method | Supabase CLI (db push) | Versionado, reproducible, project already linked |
| S2.2-D2 | Insert order for IA messages | Send first → insert with wamid | Honest data: only register what was actually sent |
| S2.2-D3 | wamid for non-Meta providers | Meta only, null for others | GOWA deprecated, format unreliable |

## Not Changed

- Frontend — message_id/delivery_status not displayed (future)
- process-campaign-queue — no wamid tracking (nullable column, works as-is)
- MassMessageDialog, ChatViewer, Inbox inserts — no wamid (Panel messages)
