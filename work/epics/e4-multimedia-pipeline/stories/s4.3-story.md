# S4.3: Edge Cases & Observability

## User Story

As a **system operator**,
I want **edge cases handled gracefully with full observability**,
so that **failures are logged, users get fallback responses, and I can debug issues from activity_logs**.

## Acceptance Criteria

### Scenario: Audio file exceeds 25MB (Whisper limit)
```gherkin
Given a user sends an audio message larger than 25MB
When transcribe-audio downloads the file and checks its size
Then the transcription is skipped
And an activity_log entry is created with type 'audio_too_large'
And the fallback message flow is triggered (bot asks user to repeat)
```

### Scenario: Invalid media URL in AI response
```gherkin
Given the AI response contains <<MEDIA:url>> with an invalid URL
When process-samurai-response extracts the media URL
Then the URL is validated with a basic format check
And if invalid, the media is skipped
And only the clean text is sent to the user
And an activity_log entry is created with type 'invalid_media_url'
```

### Scenario: Successful image send is logged
```gherkin
Given the AI response contains a valid <<MEDIA:url>>
When process-samurai-response successfully sends the image via send-message-v3
Then an activity_log entry is created with type 'image_sent'
And the log includes lead_id, media_url, and send mode (caption/split)
```

### Scenario: Media URL validation before send
```gherkin
Given the AI response contains a <<MEDIA:url>> tag
When the URL is extracted
Then a basic validation is performed (starts with https://, has valid format)
And only valid URLs proceed to the media send path
```
